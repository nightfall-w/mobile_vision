## 移动端UI自动化的三个困境

移动端UI自动化测试发展到今天，选型基本绕不开这几条路，但每条路都有各自的痛点。

**控件识别这条路，脚本比较脆弱。**

Appium、UIAutomator2 这类框架的核心思路是通过 ID、XPath 定位元素。优点是成熟稳定，缺点是页面结构变化后脚本需要同步维护。一个按钮的 ID 从
`btn_submit` 改成 `submit_btn`，或者页面层级多加了一个容器，脚本就得跟着改。回归用例集到了几百上千条时，每次版本迭代修脚本的工作量不小。

而且碰到 H5 页面、WebView、Flutter 渲染的场景，DOM 树要么不完整，要么压根拿不到，定位无从谈起。

**截图对比这条路，误报率比较高。**

部分团队尝试用 OpenCV 做截图对比来做断言，典型的如 **AirTest**（网易开源）等工具。但换一台测试机分辨率不同——误报；加载慢半秒页面状态不同——误报；改了个主题色——误报。对环境比较敏感，实践中作为主力方案有一定挑战。

**多模态大模型这条路，成本和黑盒问题也需要关注。**

这是最近一年比较热的方向——把截图丢给 GPT-5.4 Pro、Gemini 3.1 Pro、qwen3.7-plus等多模态模型，让模型"看图"
识别元素并决策操作。自然语言编用例确实方便，但实际落地后发现几个问题：

| 问题         | 表现                                       |
|------------|------------------------------------------|
| **成本偏高**    | 每张截图 800-2000+ Token，一次测试几十个步骤，大规模回归成本飙升 |
| **响应偏慢**    | 多模态模型处理图片首 Token 2-5s，步骤越多累积等待越久         |
| **精度不太稳定**  | 通用模型从像素中"猜"元素，对特定 App 的 UI 风格没有针对性优化     |
| **黑盒不易干预** | 识别错了不易定位，难以调优                             |

这些痛点驱动我去寻找另一种解法。

---

## 换个方向：让视觉引擎"看"，让大模型"想"

我换了一个思路。

**不做"让大模型看图识字"这件事。**

而是让本地视觉引擎先把页面解析清楚——页面上有什么元素、在什么位置、是什么状态、文字是什么——全部提取成结构化的 JSON，再交给
LLM 做决策。

这样做的好处：

- LLM 只需要处理几百 Token 的文本，而不是几千 Token 的图片，**成本和速度都有改善**
- 本地视觉引擎可以针对**特定 App 做小样本训练**，识别精度可以做到高于通用模型
- 整个识别过程**透明可控**，出问题可以定位、可以优化

## 核心设计：双通道页面解析

一条请求进来，系统通过两条通道获取页面信息，融合输出给 LLM。整体流程可以用下面这张图来理解：

![执行流程图](docs/%E6%89%A7%E8%A1%8C%E6%B5%81%E7%A8%8B%E5%9B%BE.png)

### 通道一：DOM 快通道（uiautomator2）

默认优先走这条路。通过 adb 调用 uiautomator2 dump 出当前页面的 XML 视图层级树。

**耗时：0.3-0.8s**

拿到的不仅是坐标和类型，还包括控件的原生交互属性：`clickable`、`enabled`、`checked`、`selected`
。按钮是否可点、复选框是否选中、输入框是否聚焦——这些信息直接从控件树中获取，**零歧义**。

### 通道二：视觉通道（YOLO + OCR + 大津法）

当遇到 WebView、Flutter、小程序等 DOM 信息不完整的页面时，自动切换至视觉通道：

1. **YOLOv8 目标检测**：识别可交互元素类型与位置
2. **OCR**：提取元素上的文字内容
3. **大津法颜色分析**：精确分析文字颜色，区分选中（绿色）、未选中（浅灰色）、禁用（灰色）等状态

下面这张图是 YOLO 模型对小程序页面的实际检测效果——搜索框、活动弹窗、关闭小程序按钮等元素被准确框出，每个框上都标注了类型和置信度：

![YOLO 模型识别效果](docs/YOLO%E6%A8%A1%E5%9E%8B%E8%AF%86%E5%88%AB%E6%95%88%E6%9E%9C.png)

可以看到，模型准确识别出了"活动弹窗"（置信度 0.95）、"弹窗关闭按钮icon"（0.93）、"客服按钮"（0.91）等元素。有了元素的位置和类型后，还需要知道它们的交互状态——这就用到颜色分析了。因为页面元素的颜色本身就承载着状态信息——导航栏菜单项选中是绿色、未选中是浅灰色；按钮置灰表示不可点击；红色文字往往是强提示。这些状态信息不是视觉效果，而是
LLM 理解当前页面逻辑的关键输入。通过大津法精确提取文字的 RGB 值和亮度，LLM 才能准确判断"主页按钮是绿色高亮状态"还是"提交按钮是灰色禁用状态"。

## 核心优势：三种方案的全方位对比

和主流的两种方案放在一起看，差异会更清楚：

| 维度               | Appium 控件识别                   | 截图 + 多模态大模型                          | MobileVision（结构化页面树）                             |
|------------------|-------------------------------|--------------------------------------|--------------------------------------------------|
| **编写/调试/排查成本**   | 写 XPath/ID 脚本，页面结构变化后需同步维护，工作量较大 | 自然语言编写成本低，但执行过程较黑盒，识别错误不易干预和复现，调试有一定难度 | 自然语言编写，思考链 + 结构化数据可回放，能辅助定位根因                    |
| **元素定位**         | 依赖 ID/XPath，页面结构微调即需修改        | 从像素中"猜"坐标，精度不太可控，且不同模型坐标格式不统一（有的返回实际坐标，有的归一化，有的等比例缩放，甚至限制截图尺寸），换模型后坐标行为可能改变 | 精确到坐标，格式固定，同一页面结果确定，不受模型影响                  |
| **LLM Token 成本** | 不涉及                           | 每张截图 800-2000+ Token，大规模回归成本高        | 结构化 JSON 仅 200-500 Token，约为多模态的 1/4 到 1/10       |
| **响应速度**         | 元素定位 ms 级，但脚本维护消耗人力           | 传图 + 多模态推理，单步操作响应约 15s，步骤越多累积等待越久 | 本地解析 DOM 0.3-0.8s / 视觉 1-2s，加 LLM 文本推理，单步操作 3-5s |
| **交互状态**         | 直接获取原生属性                      | 从像素推断，误判率较高                           | DOM 通道直接获取原生属性，视觉通道通过颜色辅助分析                             |
| **可控性**          | 高，逻辑完全可控                      | 不易定位根因和优化                         | 可追溯、可定位、可迭代优化                               |
| **针对训练**         | 不需要                           | 难以针对性训练                              | 小样本训练，300张左右截图即可训练专属模型                           |

### 几个关键差异展开说说

**关于成本。** 多模态方案的单次调用成本看起来不高，但回归测试动辄几百上千条用例，每次操作都要传一张图，累积下来成本比较可观。我的方案把页面解析放在本地完成，LLM
只处理 200-500 Token 的文本，可以用任意便宜的文本模型，甚至本地部署。

**关于可控性。** 多模态方案是一个黑盒——识别错了，很难分清是模型"没看清"还是"理解错了"，调优手段有限。

我的方案中，Agent 的每一步思考链和页面结构化数据都会完整持久化到数据库。下图为监控页面，可以看到每一步 Agent 的思考过程、当时收到的页面结构化数据、输出的操作指令，以及该步骤的执行截图，方便回溯定位问题：

![测试过程可视化](docs/%E6%B5%8B%E8%AF%95%E8%BF%87%E7%A8%8B%E5%8F%AF%E8%A7%86%E5%8C%96.png)

![思考过程输入输出分析](docs/%E6%80%9D%E8%80%83%E8%BF%87%E7%A8%8B%E8%BE%93%E5%85%A5%E8%BE%93%E5%87%BA%E5%88%86%E6%9E%90.png)

执行出错时通过回放可以辅助定位根因：是模型漏检了元素 → 优化标注数据重训；是 LLM 推理错了 → 调整 prompt 或换模型；是用例描述有歧义 → 修改用例。形成一个持续改进的闭环。

这个闭环中，模型本身的识别精度是基础。通用模型覆盖面虽广，但对特定 App 的 UI 风格不一定能精准识别。MobileVision 内置了 YOLO
训练流水线，300张左右截图 + 在线标注 + 一键训练，即可训练出针对目标 App 页面元素的专属模型，支持"商品卡片""规格弹窗""添加到购物车
Icon"等业务自定义类型。

## 实际效果

我在一个电商小程序上做了落地验证：

| 指标          | 数据                                |
|-------------|-----------------------------------|
| 训练样本        | 约 800 张截图                         |
| 自定义元素类型     | 35 个（商品卡片、规格弹窗、搜索框、Tab栏等）         |
| 模型训练时间      | 约 9小时                             |
| 识别精度（mAP50） | 0.95+                             |
| 核心元素置信度     | 稳定在 0.8 以上                        |
| 单次 Token 消耗 | 300-500 Token（仅为传图方案的 1/5 到 1/10） |
| 回归用例        | 6 条核心用例，连续 100 轮执行成功率 95%         |
| 脚本维护        | 无需维护                                 |

## 总结

MobileVision 目前还处于早期阶段，核心链路已经跑通并在实际项目中验证。相比成熟的商业方案，它还有很多不完善的地方。但我在移动端
UI 自动化测试方向上的一次明确选择：

**与其让大模型"看图说话"，不如让机器先看懂页面，再让大模型做决策。**

这条路不一定适合所有场景。如果你的测试对象经常变化、页面差异大，多模态大模型的通用性可能更有优势。但如果你长期维护固定
App 的回归测试集，希望降低长期成本、提升可控性，那么"本地视觉识别 + 文本推理 LLM"的组合值得一试。
