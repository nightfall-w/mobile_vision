## 现状与痛点

移动端 UI 自动化测试发展至今，经历了几个阶段，但每个阶段都有其难以回避的短板。

### 阶段一：传统控件识别（Appium / UIAutomator2）

这是目前绝大多数团队在用的方案——通过页面元素的 ID、XPath、Class Name 等属性进行定位和操作。

实际落地中面临的核心问题：

**脚本脆弱，维护成本高。** 一个按钮的 ID 从 `btn_submit` 改成 `submit_btn`，或者页面层级加了一个外层容器，你的脚本就挂了。在快速迭代的移动互联网团队中，这种"一改就崩"的维护成本很高。特别是回归用例集到了几百上千条的时候，每次版本迭代光修脚本就是不小的工作量。

**无法应对动态内容。** H5 页面、WebView 中的内容、Flutter/React Native 渲染的控件树，传统框架的定位能力大打折扣。对于小程序等场景，DOM 树的解析往往不完整或不准确。

### 阶段二：截图对比（OpenCV / 像素匹配）

部分团队尝试用截图对比的方式来做断言——截一张基准图，运行时再截一张，对比差异。

这个方案对环境光、分辨率、渲染时机极其敏感。换一台测试机、改一个主题色、或者网络慢了半秒导致加载状态不同，就会产生大量误报。实践中很难作为主力的自动化方案使用。

### 阶段三：多模态大模型"看图"方案

这是最近一年比较热门的方向——直接把截图丢给 GPT-4V、Claude Vision 等多模态大模型，让模型"看"图识别元素并决策操作。

听起来很理想，但我们实际调研和试跑后，发现同样存在问题：

| 问题 | 具体表现 |
|------|---------|
| **成本高昂** | 每张截图 800-2000+ Token，一次操作决策可能需要多张截图，大规模回归测试成本飙升 |
| **响应慢** | 多模态模型处理图片的首 Token 延迟 2-5s，一次测试执行几十个步骤，累积等待时间很长 |
| **精度不可控** | 通用模型从像素中"猜测"元素，对特定 App 的 UI 风格没有针对性优化，偶尔识别错完全没有规律 |
| **黑盒不可干预** | 识别错了，你没法知道是模型"没看清"还是"理解错了"，更没法针对性调优，只能反复调 prompt |
| **隐私风险** | 业务截图可能包含敏感用户数据，上传到云端 API 存在合规隐患 |

---

## 我们的思路

在分析了上述方案的问题后，我们确定了 MobileVision 的设计方向。

核心思路可以概括为一句话：**不需要多模态大模型，只需要文本推理大模型 + 本地视觉识别引擎。**

我们不做"让大模型看图识字"这件事，而是让本地视觉引擎先把页面看懂，把页面上有什么元素、在什么位置、是什么状态、文字是什么，全部提取成结构化的 JSON 数据，然后再把这个 JSON 交给 LLM 做决策。

这样做的好处很明显：
- LLM 只需要处理几百 Token 的文本，而不是几千 Token 的图片
- 本地视觉引擎可以针对特定 App 训练优化，成本低、速度快
- 整个识别过程透明可控，错了可以定位、可以调优

## 核心技术：双通道页面解析

这是 MobileVision 最核心的技术模块。一条请求进来，系统会通过两条通道获取页面信息：

### 通道一：DOM 快通道（uiautomator2）

这是默认优先使用的通道。通过 adb 调用 uiautomator2 dump 出当前页面的 XML 视图层级树。

**耗时：0.3-0.8s**

能拿到什么？不仅仅是元素的坐标和类型，还包括：
- 控件的原生交互属性：`clickable`、`enabled`、`checked`、`selected` 等
- 控件的文本内容
- 控件的精确坐标（px 级别）
- 完整的视图层级关系（父容器、子元素）

这些原生属性的价值在于：**不需要"猜"**。按钮是否可点击、复选框是否选中、输入框是否聚焦……这些信息直接从控件树中获取，零歧义。

### 通道二：视觉通道（YOLO + OCR + 大津法）

当遇到 WebView、Flutter 渲染、小程序等 DOM 信息不完整的页面时，自动切换至视觉通道：

1. **YOLOv8 目标检测**：识别页面中的可交互元素类型与位置
2. **OCR（EasyOCR / RapidOCR）**：提取元素上的文字内容
3. **大津法（Otsu）自适应二值化**：精确分析文字颜色（RGB + 亮度）

为什么需要颜色分析？因为按钮的灰色禁用态、红色错误提示、绿色成功提示，这些信息对 LLM 理解当前页面状态很重要。大津法可以精确区分文字颜色，而不是让 LLM 从像素中"感受"颜色。

### 融合输出

两条通道的输出最终融合成一个 **JSON 格式的结构化页面树**，包含：
- 页面上的所有元素列表（类型、文本、坐标、状态）
- 元素之间的空间位置关系
- 交互状态信息

这个 JSON 就是 LLM 看到的"页面"。**比截图更精确、更省 Token、更可复现。**

**LLM 从这份数据中能获取什么？**

| 信息维度 | 数据来源 | 说明 |
|---------|---------|------|
| **页面尺寸** | 截图分辨率 | `page_width: 720, page_height: 1560`，LLM 知道屏幕比例，坐标换算有基准 |
| **元素类型** | YOLO 模型识别 / DOM 通道 | 不仅知道"这里有个东西"，还知道它是"搜索Icon"还是"活动弹窗"还是"关闭按钮" |
| **精确坐标** | YOLO 检测框 / 原生属性 | 每个元素都有 `bbox: [x1, y1, x2, y2]` 和 `bbox_center`，px 级坐标，不是模糊的"大概位置" |
| **文字内容** | OCR 提取 / DOM 属性 | 直接提取"周二会员日""限时秒杀>"等文本，LLM 直接读取，无需从像素中辨字 |
| **颜色与亮度** | 大津法分析 | 文字颜色：红色（促销/强提示）、黑色（正文）、灰色（禁用），`color_brightness` 量化亮度值，可精确区分 |
| **置信度** | 模型推理结果 | 每个元素和文字都有 `confidence`，LLM 知道识别结果的可靠程度，低置信度的可交叉验证或忽略 |
| **元素关系** | 空间重叠分析 | "关闭按钮Icon"是"活动弹窗"的子元素，LLM 能理解它的作用是和弹窗绑定的 |
| **空间布局** | 坐标计算 | 搜索 Icon 在左上角、活动弹窗居中、关闭按钮在弹窗底部——LLM 能感知整体的页面布局 |

这份数据替代了"把整张截图丢给多模态模型"的方式。LLM 不需要从像素中猜测元素的坐标、文字和颜色——所有精确信息都已提取好，LLM 只需要做一件事：**基于这些信息做决策**。

这也是为什么我们的方案只需要文本推理模型——LLM 在 MobileVision 中的角色不是"看"，而是"决策"。

## Agent 执行引擎

有了结构化的页面描述，LLM 的工作就变得简单而高效。

### 执行流程

```
用户编写用例 → 编排测试计划 → 创建测试任务
                                ↓
                 Agent 读取步骤 → LLM 理解意图
                                ↓
                 双通道页面解析 → 生成结构化页面树
                                ↓
                 LLM 基于页面树 + 用例步骤推理
                 输出思考链与操作指令
                                ↓
                 ADB 执行操作（点击/输入/滑动/长按等）
                                ↓
                 验证结果 → 记录日志 → 继续下一步
```

### 可观测的思考链

Agent 执行过程中，每一步的决策依据都会以思考链（Chain-of-Thought）的形式完整输出：

- LLM 基于当前页面数据和用例步骤"为什么选择这个操作"
- 预期效果与实际情况的对比
- 最终的操作指令

这些思考链会通过 WebSocket 实时推送到监控页面，并持久化到 MySQL 数据库。用户可以在监控页面上**逐条追溯 Agent 的完整决策轨迹**。

### 动态决策步长

每次调用 LLM 规划下一步时，默认会输出**一个**动作（如"点击搜索框"），执行完后重新截图解析、再次调用 LLM 规划下一步。

但有些场景下，连续多个动作之间页面不会变化，比如：点击搜索框 → 输入关键词 → 点击搜索按钮。如果每个动作都调一次 LLM，既有延迟浪费，又增加了 Token 消耗。

MobileVision 的 Agent 会智能判断当前操作是否会引发页面变化——如果不会，就在一次 LLM 调用中规划**多个连续动作**一次性执行，跳过中间的截图解析和 LLM 调用。这个机制大约能减少 20%-30% 的页面解析和不必要的 LLM 调用开销。

## 核心优势

### 对比：传统方案 vs 多模态大模型 vs MobileVision

| 维度 | Appium 控件识别 | 截图 + 多模态大模型 | MobileVision（结构化页面树）                                                    |
|------|---------------|-------------------|-------------------------------------------------------------------------|
| **全生命周期成本**（编写/调试/排查） | 需编写 XPath/ID 定位脚本，页面结构变化即失效；定位失败报错明确但需逐条修改，维护工作量大 | 自然语言编写用例成本低，但黑盒执行，识别错误无法干预、不可复现，调试和排查全靠"猜"，反复调 prompt 效果有限 | 自然语言编写成本低；思考链 + 结构化数据完整持久化，可回放全过程，精准定位根因（模型漏检 / LLM 误判 / 用例歧义），调试和排查成本低 |
| **元素定位** | 依赖 ID/XPath，页面结构微调即需修改 | 从像素中"猜测"坐标，精度不可控，不可复现 | 精确到 px 级坐标，同一页面识别结果完全确定，可复现                                             |
| **LLM Token 成本** | 不涉及 | 每张截图 800-2000+ Token，每次操作都需传图，大规模回归成本飙升 | 结构化 JSON 仅 200-500 Token，仅为多模态方案的 1/4 到 1/10                            |
| **响应速度** | 元素定位 ms 级，但脚本维护消耗大量人力时间 | 每次操作需上传图片 + 多模态模型推理，首 Token 延迟 2-5s，操作步骤越多累积等待越久 | 本地解析 DOM 0.3-0.8s / 视觉 1-2s，无需传输图片，响应稳定且不受网络影响                          |
| **交互状态** | 直接获取原生属性（clickable / enabled / selected） | 从像素外观推断，按钮是否可点、复选框是否选中全靠"猜"，误判率高 | DOM 通道直接获取原生属性，零歧义；视觉通道通过颜色分析推断，比多模态更可靠                                 |
| **可控性** | 高，脚本逻辑完全可控 | **黑盒**，识别错误无法定位根因，无法针对性优化 | 完全可控，识别过程可追溯、错误可定位、本地模型可快速迭代优化                                          |
| **针对训练** | 不需要 | 无法针对性训练，通用模型对特定 App 的 UI 理解有限 | **小样本训练**，200张截图即可训练专属模型，精度和坐标准确度远高于通用模型                                |

### 关键差异说明

**1. 小样本训练，让模型认识你的 App**

传统多模态方案用的是通用模型，它什么都想认识，但对你 App 特有的 UI 元素往往识别不准。

MobileVision 内置了完整的 YOLOv8 训练流水线——上传几十张 App 截图，在线标注，一键训练，模型自动部署。训练出来的模型只认识你 App 的页面元素（如"商品卡片""规格弹窗""添加到购物车 Icon"等），识别精度和坐标准确度都远高于通用模型。

**2. 完全可观测、可优化**

这是本地视觉方案相比多模态方案的核心优势：
- 识别错了 → 查看结构化页面树，看是漏检还是误判
- 坐标偏了 → 调整模型或标注数据，重新训练
- LLM 理解错了 → 调整 prompt 或切换模型

形成"**执行 → 分析 → 优化 → 再执行**"的闭环，而不是在黑盒前束手无策。

**3. 专注"识得准"，而非"识得广"**

多模态大模型追求的是通用能力。MobileVision 的设计思想恰恰相反：**专注为被测 App 服务**，在垂直场景下做到更高精度。这种"小而精"的思路，在大规模回归测试中尤其有价值。

## 实际成效

目前 MobileVision 已在我们内部项目跑通全流程，针对多个电商类小程序以及下厨房app进行了落地验证：

- **模型训练**：使用约 1000 张截图，定义了 35 个自定义元素类型（商品卡片、规格弹窗、搜索框、分类按钮等），训练时间约 9小时 分钟
- **识别精度**：YOLO 模型在测试集上的 mAP50 达到 0.95+，核心元素（按钮、输入框、商品卡片）的识别置信度稳定在 0.9 以上
- **执行速度**：单步操作（页面解析 → LLM 决策 → ADB 执行）平均耗时 3-5s，主要瓶颈在 LLM 推理，页面解析本身仅 0.3-2s
- **Token 消耗**：单次决策消耗约 300-500 Token（输入：结构化页面树 + 用例步骤 + 历史记录；输出：思考链 + 操作指令），相比直接传图的方案降低 80% 以上
- **稳定性**：6 条核心回归用例，连续执行 100 轮，成功率 95%，**零脚本维护**

MobileVision 目前处于早期阶段，但核心链路已经跑通，并在实际项目中得到了验证。

我写这篇文章，不是为了说这个项目有多完善——它距离成熟还有不少距离。我更想分享的是我们在移动端 UI 自动化测试方向上的思考：**与其让大模型"看图说话"，不如让机器先看懂页面，再用小模型做精准决策。**

这条路不一定适合所有场景。如果你做的是通用 UI 测试，面对的是千奇百怪的第三方 App，那么多模态大模型可能是更好的选择。但如果你像我们一样，长期维护一个或几个固定 App 的回归测试集，追求的是低成本、高可控、可迭代的自动化方案，那么"本地视觉识别 + 文本推理 LLM"的组合值得一试。
